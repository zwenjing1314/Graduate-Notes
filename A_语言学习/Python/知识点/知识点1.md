## @ 装饰器

### @classmethod

1. 基本作用

@classmethod 装饰器用于将类方法转换为可以接收类本身作为第一个参数的方法。这个参数通常命名为 cls（约定俗成），它代表的是类对象本身，而不是实例对象。

2. 与普通实例方法的区别

```py
class MyClass:
    def instance_method(self, x):
        # self 指向具体的实例对象
        # 只能访问实例的属性和方法
        return self.value + x
    
    @classmethod
    def class_method(cls, x):
        # cls 指向类对象本身
        # 可以访问类的属性、创建新实例、调用其他类方法
        return cls.class_value + x
```

3. 调用方式的不同

```py
# 普通实例方法 - 必须通过实例调用
obj = MyClass()
obj.instance_method(10)

# 类方法 - 可以通过类名或实例调用
MyClass.class_method(10)  # ✓ 推荐方式
obj.class_method(10)       # ✓ 也可以，但 cls 仍然是 MyClass 类
```

4. 代码中的实际应用

看看你选中的 BBox.from_polygon 方法：

```py
@classmethod
def from_polygon(cls, polygon: list[float]) -> "BBox":
    """
    Convert an OCR polygon with 8 numbers into an axis-aligned bbox.
    
    Example polygon:
    [622, 138, 1137, 136, 1138, 167, 622, 168]
    """
    if len(polygon) != 8:
        raise ValueError(f"Polygon must have 8 numbers, got {len(polygon)}")

    xs = polygon[0::2]  # 提取所有 x 坐标：[622, 1137, 1138, 622]
    ys = polygon[1::2]  # 提取所有 y 坐标：[138, 136, 167, 168]
    return cls(
        x0=min(xs),  # 622
        y0=min(ys),  # 136
        x1=max(xs),  # 1138
        y1=max(ys),  # 168
    )
```

5. 与 @staticmethod 的对比

```py
class MyClass:
    @classmethod
    def with_cls(cls, x):
        # 可以访问 cls，创建实例
        return cls(x)
    
    @staticmethod
    def without_cls(x):
        # 无法访问 cls 或 self
        # 只是一个放在类命名空间下的普通函数
        return x * 2
```

### @staticmethod

静态方法装饰器

```python
class MathUtils:
    @staticmethod
    def add(a, b):
        return a + b

# 调用静态方法
result = MathUtils.add(3, 5)  # 通过类名调用
math_utils = MathUtils()
result = math_utils.add(3, 5)  # 通过实例调用
print(result)  # 输出: 8
```

### @property

内置装饰器，允许将类方法像属性一样访问，从而简化代码并增强封装性

```py
class Person:
    def __init__(self, name):
        self._name = name

    @property
    def name(self):
        return self._name

# 使用示例
person = Person("Alice")
print(person.name)  # 输出: Alice
```

### @str.setter

设置属性值，可以结合@property和@setter装饰器

```py
class Person:
    def __init__(self, name):
        self._name = name

    @property
    def name(self):
        return self._name

    @name.setter
    def name(self, value):
        if not isinstance(value, str):
            raise TypeError("name must be a string")
        self._name = value

# 使用示例
person = Person("Alice")
person.name = "Bob"  # 触发 setter 方法
print(person.name)  # 输出: Bob
```

### @str.deleter

删除属性

```py
class Person:
    def __init__(self, name):
        self._name = name

    @property
    def name(self):
        return self._name

    @name.setter
    def name(self, value):
        if not isinstance(value, str):
            raise TypeError("name must be a string")
        self._name = value

    @name.deleter
    def name(self):
        raise AttributeError("Can not delete the name")

# 使用示例
person = Person("Alice")
del person.name  # 触发 deleter 方法
```

## python中下划线操作

#### 类名前加下划线

在python中，类名前家下划线（_）是一种约定俗称的命名方式，表示该类是模块内部所用的，不建议在模块外部直接访问或导入

```py
# 在模块 my_module.py 中
class _InternalClass:
    def __init__(self):
        self.data = "内部数据"
```

#### 变量或函数名开头以_开头

它是一种非强制性的约定，表示这是一个内部使用的对象，不建议在模块或类的外部直接访问（即受保护的访问）

```py
class MyClass:
    def __init__(self):
        self._internal_var = 42  # 约定这个变量是内部使用的

    def _internal_method(self):  # 内部方法
        print("This is an internal method")

# 外部访问示例
obj = MyClass()
print(obj._internal_var)  # 可以访问，但不推荐
obj._internal_method()    # 可以调用，但这违背了约定
```

#### 变量或方法名前加双下划线

它会触发python的名称重整机制，用来避免覆盖父类中的同名属性或方法，或者避免意外修改

```py
class MyClass:
    def __init__(self):
        self.__private_var = 100  # 触发名称重整

    def __private_method(self):  # 触发名称重整
        print("This is a private method")

# 外部访问示例
obj = MyClass()

# 直接访问会报错
# print(obj.__private_var)       # AttributeError: 'MyClass' object has no attribute '__private_var'
# obj.__private_method()         # AttributeError: 'MyClass' object has no attribute '__private_method'

# 但可以通过“名称重整”的结果访问（不推荐）
print(obj._MyClass__private_var)  # 100
obj._MyClass__private_method()    # This is a private method
```

#### 单下划线_作为变量名

单下划线作为临时变量，表示不关心的变量（即变量值会被忽略）

```py
# 解包时忽略某些值
a, _, c = (1, 2, 3)
print(a, c)  # 输出: 1 3

# 遍历时不关心循环变量
for _ in range(5):
    print("Hello")  # 输出 5 次 Hello
```

#### 魔术方法/双下划线开头和结尾

这些方法是python内部的一部分，通常不需要手动调用，系统会自动调用

```py
class MyClass:
    def __init__(self, name):
        self.name = name

    def __str__(self):
        return f"MyClass object with name: {self.name}"

obj = MyClass("example")
print(obj)  # 自动调用 __str__: 输出 "MyClass object with name: example"
```

## Hook钩子

项目根目录创建文件夹**.pre-commit-config.yaml**

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-toml
      - id: check-json
      - id: check-merge-conflict
      - id: mixed-line-ending

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.11.0
    hooks:
      - id: ruff
        args: ["--fix"]
      - id: ruff-format

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.15.0
    hooks:
      - id: mypy
        additional_dependencies:
          - pydantic>=2
          - types-PyYAML
        args: ["--config-file=pyproject.toml"]
```

终端执行

```bash
pre-commit install  # 安装钩子，使检查在提交时自动触发。
pre-commit run --all-files  # 手动运行所有检查，不依赖于提交动作，用于一次性检查整个项目。
```

`pre-commit install` 是一个用于**将代码检查工具自动集成到 Git 工作流程中**的命令。它的作用是让你在每次执行 `git commit` 之前，都能自动运行预定义的代码检查（比如格式化、语法检查、类型检查等），从而保证提交的代码质量。

具体来说，这个操作做了两件事：

1. **在 Git 钩子目录下创建一个钩子脚本**
   执行 `pre-commit install` 后，pre-commit 会在当前 Git 仓库的 `.git/hooks/` 文件夹里生成一个名为 `pre-commit` 的可执行文件。这个文件就是 Git 的**预提交钩子**，它会在你每次运行 `git commit` 时被 Git 自动调用。
2. **把检查任务与提交动作绑定**
   当钩子被触发时，它会读取项目根目录下的 `.pre-commit-config.yaml` 配置文件，根据其中的定义依次执行各项检查（如运行 `black` 格式化代码、`flake8` 检查语法等）。如果所有检查通过，提交才会继续；如果失败，提交会被中断，并提示你修复问题。

### **为什么需要这一步？**

- **自动化**：无需手动运行检查脚本，提交时自动触发。
- **一致性**：团队所有成员使用相同的检查规则，保证代码风格统一。
- **早发现问题**：在代码入库前就发现潜在问题，避免把问题带到远程仓库。

## 代码质量与测试工具链

### **1. pytest** —— 自动化测试框架

- **作用**：编写和运行测试用例，确保代码功能符合预期。
- **特点**：
  - 语法简洁，支持用 `assert` 直接断言。
  - 自动发现测试文件（`test_*.py` 或 `*_test.py`）和测试函数。
  - 丰富的插件生态（如测试覆盖率 `pytest-cov`、并发测试 `pytest-xdist`）。
- **使用场景**：单元测试、集成测试、功能测试。

### **2. ruff** —— 极速代码检查与格式化工具

- **作用**：代码风格检查（linter）和自动格式化（formatter），相当于 `flake8` + `isort` + `black` 的超快替代品。
- **特点**：
  - 用 Rust 编写，速度极快，适合大型项目。
  - 可配置规则集，兼容主流规范（如 PEP 8）。
  - 能自动修复部分问题（如排序导入、移除未使用的变量）。
- **使用场景**：保证代码风格一致，避免低级语法错误。

### **3. mypy** —— 静态类型检查器

- **作用**：对使用了类型注解（Type Hints）的 Python 代码进行静态检查，提前发现类型不匹配的错误。
- **特点**：
  - 支持渐进式类型检查（可只给部分代码加类型注解）。
  - 能捕获潜在 bug（如函数参数类型错误、返回值类型不符）。
  - 与 IDE 结合，提供更好的代码补全和提示。
- **使用场景**：提高代码健壮性，尤其适合大型项目或多人协作。

### **4. pre-commit** —— Git 钩子管理工具

- **作用**：在提交代码前自动运行一系列检查（如上述的 ruff、mypy、pytest 等），阻止不符合规范的代码入库。
- **特点**：
  - 基于配置文件（`.pre-commit-config.yaml`）定义检查任务。
  - 支持多种语言和工具，可自定义钩子。
  - 检查失败时中断提交，并提示修复。
- **使用场景**：团队协作时保证所有提交的代码都经过统一的质量关卡。

### **它们如何协同工作？**

典型的开发流程中，你可能会这样使用它们：

1. **本地开发**时，用 `ruff` 格式化代码、`mypy` 检查类型、`pytest` 运行测试。
2. **提交代码前**，`pre-commit` 自动触发上述检查（可配置只检查改动的文件），若发现问题则阻止提交。
3. **CI/CD 流水线**中，再次运行这些工具，确保合并到主分支的代码依然健康。

通过这样的组合，你可以在代码生命周期的不同阶段捕获问题，既提升了个人效率，也保障了团队代码质量。
