## 进程、线程和协程

### 一、 直观比喻：工厂、工人和工作流

**进程（Process） = 一家工厂**。工厂有自己独立的厂房（内存空间）和资源（文件句柄等）。工厂和工厂之间是互相隔离的，如果想要交换物资（进程间通信 IPC），必须通过特殊的卡车和通道，成本很高。

**线程（Thread） = 工厂里的工人**。线程是真正在干活的人。一个工厂里至少有一个工人（主线程），也可以有很多工人（多线程）。工人们共享同一个厂房里的工具和材料（共享内存），所以他们协作起来很方便，但也容易发生抢夺工具引发的混乱（需要加锁）。

**协程（Coroutine） = 工人自己的“时间管理大师”工作法**。一个工人可能同时面临多个任务（比如等机器加热、等材料送达）。与其傻等（阻塞），不如在等机器加热的时候，自己主动切去处理另一批材料。这完全是工人自己的行为，不需要厂长（操作系统）来指挥。



### 二、 底层实现原理拆解

#### 1. 进程 (Process)：资源分配的基本单位

- **原理：** 当你运行一个程序时，操作系统（OS）会为它创建一个进程。OS 会为其分配虚拟内存空间、创建页表、分配文件描述符等。在 OS 内部，每个进程都有一个“身份证”叫 PCB（进程控制块），记录着进程的所有状态。
- **隔离性：** 极强。进程 A 崩溃绝对不会直接波及进程 B，因为它们的内存地址是完全物理隔离的。
- **切换成本：极大。** 操作系统要把 CPU 从进程 A 切换到进程 B 时，需要保存 A 的全部状态，加载 B 的状态，**最致命的是需要切换内存页表，导致 CPU 的缓存（TLB）大量失效**，这是一个非常昂贵的操作。

#### 2. 线程 (Thread)：CPU 调度的基本单位

- **原理：** 线程寄生在进程之中。OS 调度的最小单位其实是线程，而不是进程。同一个进程下的所有线程，共享该进程的代码段、数据段和堆内存。但是，每个线程有自己独立的**栈（Stack）**、**程序计数器（PC）\**和\**寄存器状态**（为了记住自己执行到哪行代码了）。
- **切换成本：中等。** 线程切换也是由操作系统（内核）完成的，需要陷入“内核态”。但是因为同一个进程的线程共享内存空间，所以不需要切换页表，缓存依然有效，代价比进程切换小得多。
- **痛点：** 哪怕开销小了，但如果同时开启几万个线程，OS 在调度时依然会把大量的 CPU 时间浪费在“切换上下文”上，真正干活的时间反而少了。

#### 3. 协程 (Coroutine)：用户态的轻量级线程

- **原理：** 协程完全是**用户空间（应用程序代码层面）\**的概念，操作系统内核对协程\**一无所知**。内核眼里只有线程，它不知道这个线程里其实还跑着成千上万个协程。协程本质上是一个可以“暂停执行（Yield）”并“恢复执行（Resume）”的函数。
- **调度机制：** 协作式（Cooperative）。线程是被 OS 强制剥夺 CPU 执行权的（抢占式）；而协程是依靠代码自己主动让出执行权的。比如遇到网络 I/O 阻塞时，协程 A 主动说：“我等下数据，这会儿 CPU 给协程 B 用吧”。
- **切换成本：极低。** 因为完全不需要操作系统插手，没有用户态到内核态的切换。协程切换只需要在内存里保存和恢复极少量的寄存器上下文，速度极快。开十万个协程消耗的内存往往只有几百 MB。

### 三、 核心区别总结

| **特性**     | **进程 (Process)**      | **线程 (Thread)**       | **协程 (Coroutine)**                      |
| ------------ | ----------------------- | ----------------------- | ----------------------------------------- |
| **管理者**   | 操作系统 (OS)           | 操作系统 (OS)           | 应用程序 / 语言运行时                     |
| **调度方式** | 抢占式 (时间片轮转)     | 抢占式                  | 协作式 (主动让出)                         |
| **切换开销** | 极高 (涉及内存映射切换) | 中等 (需要切换到内核态) | 极低 (纯用户态函数跳转)                   |
| **内存占用** | 大 (GB 级别)            | 中 (MB 级别栈空间)      | 极小 (KB 级别)                            |
| **并发能力** | 低 (几十个到上百个)     | 中 (几千个到上万个)     | 极高 (轻松处理十万/百万级)                |
| **多核利用** | 可以直接利用多核        | 可以直接利用多核        | 不能直接利用多核 (需要配合多进程或多线程) |

这三个概念是计算机科学里最核心的基石，也是平时开发和面试中最容易让人混淆的地方。要彻底弄懂它们，我们不仅要知道它们的定义，还要理解操作系统在底层是怎么看待它们的。

这三种技术没有绝对的好坏，只有最适合的场景：

- 如果是**CPU密集型**任务（比如视频渲染、大量数学计算），用**多进程**（避免 GIL 锁限制，利用多核）。
- 如果是传统的 GUI 应用或需要利用多核的后台服务，用**多线程**。
- 如果是**高并发的 I/O 密集型**任务（比如爬虫、高并发 Web 聊天服务器），用**协程**绝对是降维打击。
