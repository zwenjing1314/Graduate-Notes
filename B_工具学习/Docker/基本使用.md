## 安装 Docker

### 运行以下命令以卸载所有冲突包:

```bash
sudo apt remove $(dpkg --get-selections docker.io docker-compose docker-compose-v2 docker-doc podman-docker containerd runc | cut -f1)
```

### 使用 apt 仓库

1. 设置 Docker`apt`仓库。

将 Docker 的官方仓库源改为阿里云镜像源

注意：APT 里不能混用了阿里云 Docker 源和不匹配的签名，这里面的密钥和镜像源是一一对应的

```bash
# 安装必要工具（如果尚未安装）
sudo apt update
sudo apt install ca-certificates curl

# 创建存放 GPG 密钥的目录
sudo install -m 0755 -d /etc/apt/keyrings

# 下载 阿里云镜密钥
sudo curl -fsSL https://mirrors.aliyun.com/docker-ce/linux/ubuntu/gpg -o /etc/apt/keyrings/docker-aliyun.asc
sudo chmod a+r /etc/apt/keyrings/docker-aliyun.asc

# 添加阿里云镜像源（deb822 格式）
sudo tee /etc/apt/sources.list.d/docker.sources <<EOF
Types: deb
URIs: https://mirrors.aliyun.com/docker-ce/linux/ubuntu
Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
Components: stable
Signed-By: /etc/apt/keyrings/docker-aliyun.asc
EOF

sudo apt update
```

2. 安装 Docker 软件包

安装最新版本,运行:

```bash
sudo apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

3. 给 Docker daemon 配置 **registry mirror**

“把 apt 源换成阿里云 Docker 镜像”只解决了**安装 Docker 软件包**的问题，**并不会自动让 `docker run` 走国内镜像加速**。
 这两件事是分开的：

- `apt` 镜像：给你装 `docker-ce` 这些 deb 包
- Docker registry：给你拉 `hello-world`、`ubuntu`、`nginx` 这些容器镜像

先创建 daemon 配置文件

```bash
sudo mkdir -p /etc/docker
sudo tee /etc/docker/daemon.json <<EOF
{
  "registry-mirrors": [
    "https://docker.1ms.run"
  ]
}
EOF
```

然后重启 Docker：

```bash
sudo systemctl daemon-reload
sudo systemctl restart docker
```

再验证配置是否生效：

```bash
docker info
```

然后再试：

```bash
sudo docker run hello-world
```

## docker 工作原理，架构和容器化思想

可以把 Docker 理解成三层：

**最上层是“开发体验”**：你用 `docker build`、`docker run`、`docker compose up` 这些命令。
**中间层是“平台能力”**：Docker Engine 负责镜像、容器、网络、卷、API、拉取/推送镜像等。
**最底层是“内核机制”**：真正把进程隔离开、限制资源的不是 Docker 发明的新内核，而是 Linux 的 **namespaces** 和 **cgroups**。Docker 本质上是把这些底层能力产品化、工程化了。([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

先说一句最核心的话：**容器不是“小虚拟机”，而是“被隔离和限额的一组进程”**。Linux namespaces 让进程“以为”自己拥有独立的 PID、网络、挂载点、hostname 等系统视图；cgroups 则负责限制和统计 CPU、内存、IO 等资源。也就是说，容器和宿主机**共享同一个内核**，只是看见的世界和能用的资源被切分了。([man7.org](https://www.man7.org/linux/man-pages/man7/namespaces.7.html?utm_source=chatgpt.com))

### 1) Docker 到底在解决什么问题

Docker 要解决的不是“让程序跑起来”这么简单，而是：**把应用及其运行依赖，打成一个可重复、可分发、可隔离的标准包**。Docker 官方把 image 描述为一个标准化的软件包，里面包含运行容器所需的文件、二进制、库和配置；container 则是 image 的运行时实例。image 还是 **immutable（不可变）** 的，这一点非常重要，因为它让环境更可复现。([Docker Documentation](https://docs.docker.com/get-started/docker-concepts/the-basics/what-is-an-image/?utm_source=chatgpt.com))

所以容器化思想可以概括成一句话：
**把“环境”变成制品，把“部署”变成启动这个制品。**
以前你在服务器上一步步装 Java、装 Python、改配置；现在你更倾向于直接运行一个镜像，让环境随应用一起交付。Docker 官方也把它定位为优化开发工作流、容器化应用的平台。([Docker Documentation](https://docs.docker.com/get-started/?utm_source=chatgpt.com))

### 2) Docker 架构怎么理解

Docker Engine 官方定义得很清楚：它是一个 **client-server application**。主要有三部分：

- **docker CLI**：你平时敲的 `docker ...` 命令。
- **dockerd（daemon）**：长期运行的后台服务，负责管理 images、containers、networks、volumes。
- **Docker API**：CLI 和其他工具通过 API 跟 daemon 通信。([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

继续往下，`dockerd` 并不是直接手搓所有底层细节。官方文档说明它会依赖 **OCI-compliant runtime**，并且这个 runtime 是**通过 containerd** 调用的；Docker 现在也明确使用 containerd 处理容器生命周期和镜像管理。换句话说，你可以把它想成：

`docker CLI -> Docker API -> dockerd -> containerd -> OCI runtime（如 runc）-> Linux kernel`。([Docker Documentation](https://docs.docker.com/reference/cli/dockerd/?utm_source=chatgpt.com))

这条链路里每层职责不同：

- **CLI/API** 负责“表达意图”
- **dockerd** 负责“编排和管理 Docker 对象”
- **containerd / runc** 负责“真正把容器创建出来”
- **Linux kernel** 负责“真正隔离与限额”([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

### 3) `docker run nginx` 背后发生了什么

你可以按下面这个顺序理解一次启动：

1. 你执行 `docker run nginx`。CLI 把请求发给 Docker daemon。([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))
2. daemon 检查本地有没有这个 image；没有的话就从 registry 拉取。Docker 官方文档把 Docker Hub 描述为查找和分享镜像的服务，官方镜像也是常见起点。([Docker Documentation](https://docs.docker.com/get-started/docker_cheatsheet.pdf?utm_source=chatgpt.com))
3. image 会以**分层**的方式存储；现代 Docker/containerd 的 image store 用 snapshotters 管理镜像层和容器快照。([Docker Documentation](https://docs.docker.com/get-started/docker-concepts/building-images/understanding-image-layers/?utm_source=chatgpt.com))
4. 运行容器时，会基于只读镜像层，再加一层**可写层**，形成该容器自己的根文件系统视图。([Docker Documentation](https://docs.docker.com/get-started/docker-concepts/building-images/understanding-image-layers/?utm_source=chatgpt.com))
5. 底层 runtime 创建 namespaces，让这个进程组看到自己的 PID、网络、挂载、hostname 等；再挂上 cgroups，对 CPU、内存、IO 做限制和统计。([man7.org](https://www.man7.org/linux/man-pages/man7/namespaces.7.html?utm_source=chatgpt.com))
6. 网络、卷、端口映射等也在这一步配置好，然后容器的主进程启动，容器就“活”了。Docker 的 `run` 文档和 Engine 文档都把网络、存储、日志等作为容器运行时的重要部分。([Docker Documentation](https://docs.docker.com/engine/containers/run/?utm_source=chatgpt.com))

所以从本质上说，**Docker 不是在“虚拟出一台机器”，而是在“启动一个带独立视图的进程世界”**。这就是它通常比虚拟机更轻、更快的根本原因：它不需要每个实例都带一整套 guest OS 内核。这个结论直接来自“容器依赖 Linux namespaces/cgroups，而不是完整虚拟硬件”的架构事实。([man7.org](https://www.man7.org/linux/man-pages/man7/namespaces.7.html?utm_source=chatgpt.com))

### 4) 镜像、容器、卷、网络，分别是什么

理解 Docker，最容易混淆的是这四个对象：

- **Image**：静态模板，像“类”或“安装包”。不可变、可分层。([Docker Documentation](https://docs.docker.com/get-started/docker-concepts/the-basics/what-is-an-image/?utm_source=chatgpt.com))
- **Container**：image 的运行时实例，像“对象”或“进程”。可以创建、启动、停止、删除。([Docker Documentation](https://docs.docker.com/get-started/docker_cheatsheet.pdf?utm_source=chatgpt.com))
- **Volume**：把数据放到容器生命周期之外，否则删容器时可写层数据很容易一起消失。Docker Engine 的存储文档把 volume 作为持久化容器数据的主要方式。([Docker Documentation](https://docs.docker.com/engine/daemon/?utm_source=chatgpt.com))
- **Network**：让容器之间、容器与宿主机之间通信。Docker Engine 文档明确把网络管理作为核心能力之一。([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

一个很实用的心智模型是：
**image 管代码和依赖，container 管运行态，volume 管数据，network 管连通性。**

### 5) Docker 和虚拟机，到底差在哪

最核心的差别是：

- **虚拟机**：虚拟的是整台机器，通常每个实例有自己的 guest OS 内核。
- **容器**：隔离的是进程空间，多个容器共享宿主机内核。([man7.org](https://www.man7.org/linux/man-pages/man7/namespaces.7.html?utm_source=chatgpt.com))

这也解释了两个常见现象：

第一，Linux 上跑 Linux 容器最“原生”，因为底层直接就是 Linux 内核能力。
第二，在非 Linux 主机上，跑 Linux 容器通常要借助虚拟化层。官方文档里，Windows 上的 Docker Desktop 支持 **WSL 2 based engine**；macOS 则有自己的 **Virtual Machine Manager**。所以你在 Mac/Windows 上感觉像“直接跑 Docker”，底层其实常常仍有一层 Linux 环境。([Docker Documentation](https://docs.docker.com/desktop/features/wsl/?utm_source=chatgpt.com))

### 6) 容器化思想到底该怎么理解

我觉得有 4 个关键词最重要：

**第一，可复现。**
同一个 image 在不同环境中更容易保持一致。Docker 的资料把 image/container 的可移植性和标准化作为核心概念。([Docker Documentation](https://docs.docker.com/get-started/docker_cheatsheet.pdf?utm_source=chatgpt.com))

**第二，不可变基础设施。**
不要 SSH 进服务器临时改环境，而是改 Dockerfile，重新构建 image，再重新发布。因为 image 是不可变的，这种方式更可追踪。([Docker Documentation](https://docs.docker.com/get-started/docker-concepts/the-basics/what-is-an-image/?utm_source=chatgpt.com))

**第三，进程级隔离。**
容器的本质不是“打包文件夹”，而是“给进程一个受控边界”。边界来自 namespaces、cgroups、挂载、能力权限等。([man7.org](https://www.man7.org/linux/man-pages/man7/namespaces.7.html?utm_source=chatgpt.com))

**第四，状态外置。**
容器本身应尽量短暂、可替换；持久数据放 volume、对象存储或外部数据库。这和 Docker 把卷作为专门对象的设计是一致的。([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

### 7) 初学者最容易误解的几点

**误解一：Docker = 虚拟机。**
不对。容器更接近“被隔离的进程组”。([man7.org](https://www.man7.org/linux/man-pages/man7/namespaces.7.html?utm_source=chatgpt.com))

**误解二：有 Docker 就自动安全。**
不对。Docker 提供隔离边界，但安全还取决于镜像来源、运行权限、挂载、capabilities、是否 rootless 等配置。Docker Engine 文档专门提供 rootless mode 和安全相关配置。([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

**误解三：容器里应该像小服务器一样手工维护。**
不推荐。更符合容器化思想的是：容器可替换，修改通过新 image 发布，而不是登录进去“养”它。这个思路来自 image 不可变和构建驱动的工作流。([Docker Documentation](https://docs.docker.com/get-started/docker-concepts/the-basics/what-is-an-image/?utm_source=chatgpt.com))

### 8) 一个最简洁的心智模型

你可以把 Docker 想成：

- **Dockerfile**：配方
- **Image**：烤好的标准半成品
- **Container**：把半成品加热后真正上桌
- **Registry**：仓库
- **Volume**：冰箱/储物间
- **Network**：餐厅里的传菜系统
- **dockerd + containerd + runc**：后厨流水线
- **namespaces + cgroups**：厨房隔断和配额制度([Docker Documentation](https://docs.docker.com/engine/?utm_source=chatgpt.com))

你一旦抓住这句话，Docker 就不神秘了：

**Docker 的价值，不在于“发明了容器”，而在于把镜像构建、分发、运行、隔离、网络、存储、开发工作流整合成了一套统一体验。**([Docker Documentation](https://docs.docker.com/get-started/?utm_source=chatgpt.com))
