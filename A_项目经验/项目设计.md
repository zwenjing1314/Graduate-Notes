# ocr_inspector

## job_skeleton.py 文件

本质上是在定义“一个 OCR 任务的标准骨架”，不是做识别算法的，而是给一次任务分配统一的目录、文件路径和少量公共规则。

###   作用

  在 job_skeleton.py:26 它先做上传类型识别，判断文件是 pdf 还是 image。在 job_skeleton.py:38 它定义了 JobSkeleton，把这次任务会用到的路径都
  集中起来，比如上传目录、输出目录、pages/、overlays/、texts/、markdown/、analysis/，以及 ocr.json、full_text.txt、document.md、
  job_manifest.json 这些产物路径。在 job_skeleton.py:56 的 create() 里，它会生成 job_id、创建目录；在 job_skeleton.py:103 的 output_url()
  里，又把磁盘路径统一转成前端可访问的 /outputs/{job_id}/... URL。

  app.py 里实际就是把它当“任务上下文”来用：先创建骨架 app.py:318，再把上传文件存到 job.source_path app.py:330，然后把 job.output_dir 交给
  OCR 主流程 app.py:363，最后再用它生成分析页、manifest 和下载链接 app.py:377。



### 为什么这样设计

原因很简单：把“任务目录结构”和“业务流程”拆开。否则这些路径字符串会散落在 app.py、ocr_engine.py、分析页、前端响应里，到处都是 output_dir /
  "pages"、output_dir / "ocr.json" 这种硬编码，后面一改结构就要全项目找。现在集中到一个文件里，相当于有了单一事实来源。

### 这个设计还有几个直接好处：

  - 扩展新产物很方便。比如后来新增 document.md，就是先在骨架里加 document_markdown_path，再在接口和 manifest 暴露出去。项目笔记也明确这么写
    了：notes/modif_record.md:156。
  - 分层更清楚。ocr_engine.py 负责“怎么做 OCR”，app.py 负责“怎么接请求和组织流程”，job_skeleton.py 负责“这次任务的文件系统长什么样”。笔记里
    把它叫“任务与产物层”，并且说它是“后续复用骨架”：notes/stu_record.md:82。
  - 类型更明确。这里用 dataclass(slots=True)，说明作者希望它是一个固定字段的轻量结构体，不是随便塞键值的 dict。

## 一次 /ocr 请求从上传到产物落盘”的完整流程

接收上传 -> 判断类型 -> 创建任务骨架 -> 落原文件 -> 跑 OCR 主流程 -> 写各类产物 -> 生成分析页和 manifest -> 返回可访问 URL

  1. 请求进入后，先做输入校验
    在 app.py:300 开始接收表单和文件。先检查有没有文件名，然后调用 job_skeleton.py:26 的 detect_upload_kind()，根据扩展名和 content_type 把上
    传判定成 pdf 或 image。

  2. 立刻创建这次任务的“骨架”
    接着在 app.py:318 调用 job_skeleton.py:56 的 JobSkeleton.create()。这一步会：

  - 生成一个 job_id
  - 创建 uploads/<job_id>/
  - 创建 outputs/<job_id>/
  - 预建 pages/、overlays/、texts/、markdown/、analysis/
  - 约定好这次任务所有关键文件路径，比如 source_path、ocr.json、full_text.txt、document.md、job_manifest.json

  也就是说，从这一步开始，这次任务的磁盘位置已经固定了。

  3. 把上传文件真正写到磁盘
    在 app.py:330 会调用 _save_upload()，把用户上传内容流式写到 job.source_path。这个路径就是骨架里定义好的 uploads/<job_id>/source.xxx，保存
    时还顺手算 SHA-256，用来做请求去重和缓存指纹 app.py:112。

  4. 做缓存和并发控制
    写完原文件后，app.py 会把“文件哈希 + OCR 参数”组成 request_fingerprint，然后检查：

  - 有没有完全相同请求的缓存结果
  - 当前是否已有别的 OCR 在跑
  - 相同请求是否已经在处理中

  这部分在 app.py:331 到 app.py:360。如果命中缓存或并发冲突，这次新建的骨架目录会被清掉，不会继续生成新产物。

  5. 进入 OCR 主流程，开始在 outputs/<job_id>/ 下产出文件
    真正做识别的是 ocr_engine.py:1430 的 run_ocr_pipeline()，它拿到的正是：

  - source_path=job.source_path
  - output_dir=job.output_dir

  先确保 pages/、overlays/、texts/、markdown/ 存在 ocr_engine.py:1448，然后用 prepare_page_images() 把源文件转成按页图片
  ocr_engine.py:1457：

  - 如果是 PDF，就渲染成 pages/page_001.png、page_002.png...
  - 如果是单张图片，也会整理成 page 形式

  6. 每一页分别做 OCR，并写页面级产物
    在 ocr_engine.py:1467 开始逐页处理：

  - _ocr_image() 做识别，得到 words、lines、text
  - _draw_overlay() 生成叠框图 overlays/page_001_overlay.png ocr_engine.py:1481
  - 把纯文本写到 texts/page_001.txt ocr_engine.py:1490
  - 先记下 markdown/page_001.md 这个路径名，稍后统一写 ocr_engine.py:1498

  这一轮结束后，内存里已经有完整的 pages 数据结构。

  7. 页面都跑完后，写文档级产物
    在 ocr_engine.py:1504 到 ocr_engine.py:1564：

  - 拼接所有页面文本，写 full_text.txt
  - 调 _analyze_document_layout() 做版面分析
      - 页眉页脚过滤
      - 标题识别
      - 段落拼接
      - 列表识别
      - 阅读顺序整理
  - 给每页写结构化 Markdown：markdown/page_001.md
  - 给整份文档写 document.md
  - 最后把整份 OCR 结果序列化成 ocr.json

  这里 ocr.json 是最完整的机器可读结果，document.md 是更偏“人读”的整理结果。

  8. 回到 app.py，补写分析页和任务清单
    OCR 主流程结束后，app.py 继续做两件事：

  - 写错误分析页 analysis/index.html app.py:377
  - 写 job_manifest.json，把这次任务有哪些产物、各自 URL 是什么记录下来 app.py:81

  这里 URL 的生成统一走 job_skeleton.py:103 的 output_url()，把磁盘路径变成 /outputs/<job_id>/... 这种前端能访问的地址。

  9. 最后把可访问结果返回给前端
    在 app.py:202 构造响应 JSON，里面会带：

  - downloads.ocr_json
  - downloads.full_text
  - downloads.document_markdown
  - downloads.analysis_page
  - downloads.job_manifest
  - 每页的预览图、叠框图、文本、Markdown 地址

  然后在 app.py:390 返回给前端。

  成功落盘后，目录大致长这样

```
  uploads/<job_id>/
    source.pdf

  outputs/<job_id>/
    ocr.json
    full_text.txt
    document.md
    job_manifest.json
    pages/
      page_001.png
    overlays/
      page_001_overlay.png
    texts/
      page_001.txt
    markdown/
      page_001.md
    analysis/
      index.html
```

  所以你可以把 job_skeleton.py 理解成这条链路里的“任务文件系统合同”：

  - app.py 用它决定这次任务该把东西放哪
  - ocr_engine.py 用它给定的 output_dir 写产物
  - analysis_report.py 和接口响应再基于这些路径暴露下载地址
