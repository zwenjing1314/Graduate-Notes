# Git + 合适的协作流程 + 代码评审 + CI/CD.

在现代软件工程中，**CI/CD** 是让你的项目能够“自动化运转”的超级流水线。简单来说，它们的核心目的是：**把繁琐的测试、打包、发布工作交给机器自动完成，让你只专注写代码。**

结合你之前提到的 FastAPI 和多个小 Demo 的项目，我来为你拆解这两个概念：

### 1. CI (Continuous Integration - 持续集成)

**“持续集成”指的是：频繁地将代码合并到主干（`main` 分支），并且每次合并前都要进行自动化的测试和构建。**

- **它解决什么问题？** 假设你写了一个“数据清洗”的小模块，准备合并到主分支。你可能忘了这个改动会影响之前的 YOLO 目标检测模块，导致主分支崩溃。这就是著名的“集成地狱”。
- **它是怎么工作的？** 当你在 Git 提交代码（比如发起一个 Pull Request）时，CI 服务器（比如 GitHub Actions 或 GitLab CI）会立即启动一台虚拟机器：
  1. 自动拉取你的代码。
  2. 自动安装你 `forgeflow` 环境里的依赖（如 FastAPI, PyTorch）。
  3. **自动运行你写好的测试脚本**（比如使用 `pytest` 检查接口是否返回 200 OK）。
  4. 如果测试失败，它会立刻亮红灯，阻止你合并到主分支；如果全绿，才允许合并。

### 2. CD (Continuous Delivery / Continuous Deployment - 持续交付 / 持续部署)

CD 是 CI 的下一步。当代码成功合并到主分支后，CD 负责将这些代码安全地送到用户面前。

- **Continuous Delivery（持续交付）：** 代码经过 CI 测试后，自动打包成可以随时发布的状态（比如打包成一个 Docker 镜像，或者生成一个可以直接运行的执行文件）。但是，**最终是否上线，需要人工点击按钮确认**。
- **Continuous Deployment（持续部署）：** 更进一步，连人工确认的步骤都省了。只要代码合并到了 `main` 分支，且通过了所有自动化测试，系统就会**自动把新代码发布到你的生产服务器上**。

------

### 💡 在你的 FastAPI 项目中，CI/CD 跑起来是什么样子的？

想象一下你未来的开发日常，当配置好 CI/CD 后，流程会非常丝滑：

1. **开发（Git）：** 你在本地新建分支 `feat/new-demo`，写完了一个新的 FastAPI 路由接口，在终端敲下 `git push`。
2. **CI 介入（自动化测试）：** GitHub Actions 监听到你的 Push，自动运行脚本：
   - 检查代码格式（比如用 `flake8` 或 `black`，代码不规范直接打回）。
   - 运行单元测试，确保你的新接口和旧接口都能正常工作。
3. **代码评审（Code Review）：** 测试全绿后，你的朋友或同事（或者你自己检查一遍）确认逻辑没问题，点击 Merge 合并到 `main` 分支。
4. **CD 介入（自动化部署）：** `main` 分支一更新，服务器自动触发部署：
   - 服务器自动拉取最新代码。
   - 自动重启 FastAPI 的 `uvicorn` 进程。
5. **结果：** 你只要写完代码并提交，几分钟后，你刷新网页，你的新功能就已经完美上线了，期间你没有碰过一次服务器。

对于你这个长期项目，第一步先把 Git 和本地开发弄熟，等项目稍微成型（有了几个稳定的小模块后），再去引入 GitHub Actions 这样的工具来写 CI/CD 脚本，会让你的开发体验产生质的飞跃。
